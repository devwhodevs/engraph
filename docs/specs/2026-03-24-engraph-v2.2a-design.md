# engraph v2.2a — Context Engine

**Date:** 2026-03-24
**Milestone:** v0.4
**Depends on:** v0.3.0 (vault graph, graph agent, 3-lane RRF)
**Spec:** Subset of `2026-03-24-engraph-v2-design.md` section 7 (Context Engine)

---

## Goal

Add a context engine that transforms raw search results and vault data into agent-ready context bundles. Six library functions exposed as CLI subcommands. No server, no embedding — pure text assembly from SQLite and disk.

## What's in scope

1. `read(file_or_docid)` — full note content + metadata
2. `list(folder?, tags?, limit?)` — list notes by metadata filters
3. `vault_map()` — vault structure overview
4. `who(name)` — person context bundle
5. `project(name)` — project context bundle
6. `context(topic, max_chars?)` — rich topic bundle with budget trimming
7. CLI: `engraph context <subcommand>` with `--json`

## What's NOT in scope (deferred)

- MCP server / stdio transport → v0.4b (v2.2b)
- HTTP daemon / REST API → v0.5
- Auth / agent permissions → v0.5
- Write operations (create, append, update_metadata, link) → v0.5 (v2.3)
- Context descriptions table → v0.7 (v2.5)
- Query expansion / reranker → v0.6 (v2.4)

## 1. Architecture

New module `src/context.rs` — pure functions taking `&Store` and vault path, returning structured data. No I/O beyond file reads, no model loading, no embedding.

All functions read full note content from **disk** (raw `.md` files), not from SQLite chunks. SQLite provides metadata, search, and graph queries. Agents want full note content, not chunked snippets.

Character approximation for token budgets: ~4 chars per token. Default budget: 32000 chars (~8000 tokens).

All output structs derive `Serialize` for `--json` support.

### Shared Context Parameter

All context functions receive a shared parameter struct instead of raw args:

```rust
/// Shared context for all context engine functions.
/// Constructed once in the CLI handler, passed to each function.
pub struct ContextParams<'a> {
    pub store: &'a Store,
    pub vault_path: &'a Path,       // from store.get_meta("vault_path")
    pub data_dir: &'a Path,         // ~/.engraph/
    pub profile: Option<&'a VaultProfile>,  // from Config::load_vault_profile()
}
```

For `context(topic)` which needs search, the CLI handler loads `Embedder` + `HnswIndex` from `data_dir` (same as `engraph search` does) and passes them alongside `ContextParams`. This pays the model load cost once. Other functions (`read`, `list`, `vault_map`, `who`, `project`) do not need the embedder.

### Data Flow

```
CLI (main.rs)
  │ loads Store, vault_path, profile, (optionally Embedder + HnswIndex)
  │ constructs ContextParams
  ▼
context.rs (pure functions)
  │
  ├── Store (SQLite: files, edges, FTS, chunks)
  ├── vault_path (disk: raw .md files)
  ├── profile (vault type, structure, folders)
  └── search::search_internal (only for context topic)
```

## 2. Functions

### `read(file_or_docid) -> NoteContent`

Return full note content with metadata.

```rust
pub struct NoteContent {
    pub path: String,
    pub docid: Option<String>,
    pub content: String,         // raw .md file content
    pub tags: Vec<String>,
    pub frontmatter: String,     // raw YAML block (if any)
    pub body: String,            // content without frontmatter
    pub outgoing_links: Vec<String>,  // files this note links TO via wikilinks
    pub incoming_links: Vec<String>,  // files that link TO this note (backlinks)
    pub mentions_people: Vec<String>, // people this note mentions (outgoing mention edges)
    pub mentioned_by: Vec<String>,    // notes that mention this note as a person (incoming mention edges)
    pub char_count: usize,
}
```

**Resolution** (same as `engraph graph show`):
1. If starts with `#` and is 7 chars → docid lookup
2. Exact path match
3. Case-insensitive basename match

**File reading:** Resolve path relative to vault root (from `store.get_meta("vault_path")`). Read raw file from disk.

### `list(filters) -> Vec<NoteListItem>`

List notes by metadata. No semantic search — pure SQL filtering.

```rust
pub struct ListFilters {
    pub folder: Option<String>,    // path prefix filter
    pub tags: Vec<String>,         // require all listed tags
    pub limit: usize,              // default 20
}

pub struct NoteListItem {
    pub path: String,
    pub docid: Option<String>,
    pub tags: Vec<String>,
    pub indexed_at: String,
    pub edge_count: usize,         // total edges (wikilinks + mentions)
}
```

**SQL:**
- Folder filter: `WHERE path LIKE ?1 || '%'`
- Tag filter: `AND EXISTS (SELECT 1 FROM json_each(tags) WHERE value = ?)`  (for each tag)
- Order by: `indexed_at DESC`
- Limit: `LIMIT ?`

### `vault_map() -> VaultMap`

Vault structure overview for agent orientation.

```rust
pub struct VaultMap {
    pub vault_path: String,
    pub vault_type: String,        // "Obsidian", "Logseq", "Plain"
    pub structure: String,         // "Para", "Folders", "Flat"
    pub total_files: usize,
    pub total_chunks: usize,
    pub total_edges: usize,
    pub folders: Vec<FolderInfo>,
    pub top_tags: Vec<(String, usize)>,  // (tag, count) top 20
    pub recent_files: Vec<String>,       // 10 most recently indexed
}

pub struct FolderInfo {
    pub path: String,
    pub note_count: usize,
}
```

**Implementation:**
- Use `ContextParams.profile` for vault type/structure (if available, else defaults)
- Query `files` table grouped by folder prefix for note counts
- Query `files.tags` via `json_each` aggregation for top tags
- Query `files` ordered by `indexed_at DESC LIMIT 10` for recent files
- Include edge stats from `get_edge_stats()`

### `who(name) -> PersonContext`

Person context bundle.

```rust
pub struct PersonContext {
    pub name: String,
    pub note: Option<NoteContent>,     // the person's note (if found)
    pub mentioned_in: Vec<MentionInfo>, // notes that mention this person
    pub linked_from: Vec<String>,       // notes linking TO this person via wikilinks
    pub linked_to: Vec<String>,         // notes this person links to
    pub total_chars: usize,
}

pub struct MentionInfo {
    pub path: String,
    pub docid: Option<String>,
    pub snippet: String,          // line(s) containing the person's name
}
```

**Snippet extraction for `who()`:** Use `store.fts_search(name, limit)` to get FTS snippets for the person's name. FTS5 already highlights matches with `<b>...</b>` tags. If FTS returns no results (name not in FTS index as a standalone term), fall back to reading the mentioning file from disk and finding the first line containing the name (case-insensitive `str::to_lowercase().contains()`). Truncate to 200 chars.

```rust
// Pseudocode for snippet extraction
fn get_mention_snippet(store: &Store, file_id: i64, name: &str, vault_path: &Path) -> String {
    // Try FTS first
    if let Ok(fts_results) = store.fts_search(name, 1) {
        if let Some(r) = fts_results.iter().find(|r| r.file_id == file_id) {
            return r.snippet.clone();
        }
    }
    // Fallback: read file from disk, find first line containing name
    if let Some(path) = store.get_file_path_by_id(file_id).ok().flatten() {
        let full_path = vault_path.join(&path);
        if let Ok(content) = std::fs::read_to_string(&full_path) {
            let name_lower = name.to_lowercase();
            for line in content.lines() {
                if line.to_lowercase().contains(&name_lower) {
                    return truncate(line, 200);
                }
            }
        }
    }
    String::new()
}
```

**Implementation:**
1. Find person note: search `files` table for paths containing the name (case-insensitive basename match in People folder, or alias match via frontmatter)
2. If found, read full content via `read()`
3. Get incoming mention edges: `store.get_incoming(person_file_id, Some("mention"))`
4. For each mention, get file path and best snippet
5. Get wikilink edges: `store.get_incoming(person_file_id, Some("wikilink"))` and `get_outgoing`

**Name resolution:**
- Exact match on filename (strip `.md`)
- Fallback: scan people folder files for alias match in frontmatter
- If no People folder configured, search all files by basename

### `project(name) -> ProjectContext`

Project context bundle.

```rust
pub struct ProjectContext {
    pub name: String,
    pub note: Option<NoteContent>,      // the project note
    pub child_notes: Vec<NoteListItem>, // notes in same folder or linking to project
    pub active_tasks: Vec<TaskItem>,    // unchecked checkboxes from project + children
    pub team: Vec<String>,              // people notes linked from project
    pub recent_mentions: Vec<MentionInfo>, // daily notes mentioning the project
    pub total_chars: usize,
}

pub struct TaskItem {
    pub text: String,
    pub source_file: String,
}
```

**Implementation:**
1. Find project note: exact path or basename match, prefer notes in `01-Projects/`
2. Read full content
3. Child notes: files in the same folder + files with wikilinks pointing to the project note
4. Active tasks: scan project note + child notes for `- [ ] ` lines
5. Team: follow outgoing wikilinks from project note, filter to People folder
6. Recent mentions: search FTS for project name, filter to `07-Daily/` folder, limit 5

### `context(topic, max_chars?) -> ContextBundle`

Rich topic bundle — the flagship function. Assembles a comprehensive context within a character budget.

```rust
pub struct ContextBundle {
    pub topic: String,
    pub sections: Vec<ContextSection>,
    pub total_chars: usize,
    pub budget_chars: usize,
    pub truncated: bool,
}

pub struct ContextSection {
    pub label: String,           // "Search Result", "Related (1-hop)", "Person", "Project"
    pub path: String,
    pub docid: Option<String>,
    pub content: String,         // full or trimmed note content
    pub relevance: String,       // "direct match", "1-hop from X", "mentioned person"
}
```

**Implementation:**
1. Run hybrid search for topic (reuse existing `run_search` logic but return structured results instead of printing)
2. Take top 5 results
3. For each, read full note content from disk
4. Follow graph: get 1-hop wikilink neighbors of top results
5. Detect mentioned people in top results, include their person notes
6. Detect parent project/area (check if note is inside a Projects subfolder, include the parent folder's index note if it exists)
7. Assemble sections in priority order, trimming or dropping sections when budget exhausted:
   - Priority 1: Top search results (full content)
   - Priority 2: People notes mentioned in results
   - Priority 3: Graph-expanded notes (1-hop, full content)
   - Priority 4: Parent project note
8. Each note's content is trimmed to `max_chars / expected_sections` if the full content would bust the budget

**Budget management:**
- Default: 32000 chars (~8000 tokens)
- Section header overhead: ~100 chars per section
- If total assembled content exceeds budget, trim lowest-priority sections first
- If a single note exceeds half the budget, truncate it with `... [truncated, full note: #docid]`

### Internal Search Function

`context(topic)` needs to run a search without printing to stdout. Extract the search logic from `search.rs:run_search` into a reusable internal function:

```rust
// In search.rs
pub struct InternalSearchResult {
    pub file_path: String,
    pub file_id: i64,
    pub score: f64,
    pub heading: Option<String>,
    pub snippet: String,
    pub docid: Option<String>,
}

/// Run hybrid search and return structured results (no I/O).
/// Used by both run_search (CLI) and context engine.
pub fn search_internal(
    query: &str,
    top_n: usize,
    store: &Store,
    embedder: &mut Embedder,
    index: &HnswIndex,
) -> Result<Vec<InternalSearchResult>>
```

Refactor `run_search` to call `search_internal` then format output. This avoids duplicating the 3-lane RRF logic.

**IMPORTANT:** `context(topic)` needs an `Embedder` and `HnswIndex` for the search step. These must be passed in by the caller (CLI handler in main.rs). The `context.rs` module itself does NOT load models — it receives them as parameters.

**Type conversion:** `InternalSearchResult` and `RankedResult` share the same fields. Add a `From<InternalSearchResult> for RankedResult` impl so `context()` can feed search results into `graph_expand` without manual mapping:

```rust
impl From<InternalSearchResult> for RankedResult {
    fn from(r: InternalSearchResult) -> Self {
        RankedResult {
            file_path: r.file_path,
            file_id: r.file_id,
            score: r.score,
            heading: r.heading,
            snippet: r.snippet,
            docid: r.docid,
        }
    }
}
```

**Error handling — file not found on disk:** When `read()` or any content-loading function finds a file in SQLite but it's been deleted from disk since last index, return the metadata without content: `NoteContent { content: String::new(), body: "[File not found on disk. Re-run 'engraph index' to update.]".into(), ... }`. Do not error — the caller gets partial data and can decide what to do.

## 3. CLI: `engraph context`

```
engraph context read <file_or_docid>
engraph context list [--folder <path>] [--tags <t1,t2>] [--limit N]
engraph context vault-map
engraph context who <name>
engraph context project <name>
engraph context topic <query> [--budget <chars>]
```

All support `--json` (global flag already exists).

**Human output:** formatted text with headers and indentation.
**JSON output:** serialized structs.

## 4. Store Additions

New query methods on `Store`:

```rust
/// List files filtered by folder prefix and/or tags.
pub fn list_files(&self, folder: Option<&str>, tags: &[String], limit: usize) -> Result<Vec<FileRecord>>

/// Get note counts per top-level folder.
/// Groups by the first path component (before first `/`).
/// Files at vault root are grouped under "(root)".
/// SQL: SELECT CASE WHEN instr(path,'/') > 0 THEN substr(path,1,instr(path,'/')-1) ELSE '(root)' END AS folder, COUNT(*) FROM files GROUP BY folder ORDER BY COUNT(*) DESC
pub fn folder_note_counts(&self) -> Result<Vec<(String, usize)>>

/// Get top N tags by frequency across all files.
pub fn top_tags(&self, limit: usize) -> Result<Vec<(String, usize)>>

/// Get N most recently indexed files.
pub fn recent_files(&self, limit: usize) -> Result<Vec<FileRecord>>

/// Count edges for a specific file.
pub fn edge_count_for_file(&self, file_id: i64) -> Result<usize>
```
