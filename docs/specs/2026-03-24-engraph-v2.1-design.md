# engraph v2.1 — Graph & Graph Agent

**Date:** 2026-03-24
**Milestone:** v0.3
**Depends on:** v0.2.0 (hybrid search, smart chunking, vault profiles)
**Spec:** Extracted from `2026-03-24-engraph-v2-design.md` sections 2 (Vault Graph) and 5.3 (Graph Agent)

---

## Goal

Add a vault graph built during indexing and a graph search agent that expands search results by following connections. This creates a third RRF lane that surfaces related notes the semantic and keyword lanes miss.

## What's in scope

1. Vault graph — `wikilink` and `mention` edges stored in SQLite, built during indexing
2. Graph agent — 3rd search lane that expands seed results by following graph connections
3. 3-lane RRF integration
4. `engraph graph` CLI subcommand

## What's NOT in scope (deferred)

- Tag edges in `edges` table → tag overlap queried via JSON, no edge storage needed
- Folder edges → queryable from `files.path`, no edge storage needed
- Context engine (`context()`, `who()`, `project()`) → v0.4 (ships with MCP server)
- Token-budgeted bundles → v0.4
- Folder centroids → v0.5
- Temporal agent → v0.6
- Query classification / research orchestrator → v0.6

## 1. Vault Graph

### Schema

New table added via `migrate()` in `store.rs` (NOT in the SCHEMA constant — added dynamically like the docid migration, using `CREATE TABLE IF NOT EXISTS` after checking `sqlite_master`):

```sql
CREATE TABLE IF NOT EXISTS edges (
    id         INTEGER PRIMARY KEY,
    from_file  INTEGER NOT NULL REFERENCES files(id) ON DELETE CASCADE,
    to_file    INTEGER NOT NULL REFERENCES files(id) ON DELETE CASCADE,
    edge_type  TEXT NOT NULL,  -- 'wikilink' or 'mention'
    UNIQUE(from_file, to_file, edge_type)
);

CREATE INDEX IF NOT EXISTS idx_edges_from ON edges(from_file);
CREATE INDEX IF NOT EXISTS idx_edges_to ON edges(to_file);
CREATE INDEX IF NOT EXISTS idx_edges_type ON edges(edge_type);
```

**v2.1 edge types (only two):**
- `wikilink` — `[[Note A]]` in file B creates bidirectional edges: B → A and A → B.
- `mention` — file mentions a person detected from the People folder.

**Tag overlap** is queried via the existing `tags` JSON column on `files`:

```sql
SELECT DISTINCT f2.id, f2.path
FROM files f1
JOIN files f2 ON f2.id != f1.id
WHERE f1.id = ?
AND EXISTS (
    SELECT 1 FROM json_each(f1.tags) t1
    JOIN json_each(f2.tags) t2 ON t1.value = t2.value
)
```

**Folder membership** is queryable from `files.path` — no edges needed.

**ON DELETE CASCADE** on both `from_file` and `to_file` means deleted files automatically clean up all their edges in both directions. No manual cleanup needed for the file deletion path.

### Wikilink Extraction

New function `extract_wikilink_targets(text: &str) -> Vec<String>` — adapted from the byte-scanning approach in `profile.rs:count_wikilinks`, but returns the captured target names instead of just counting.

Extraction runs on **raw file content** (not chunk text) to avoid deduplication issues from overlapping chunks. The `UNIQUE(from_file, to_file, edge_type)` constraint handles any remaining duplicates via `INSERT OR IGNORE`.

Steps during indexing:
1. For each file being indexed, call `extract_wikilink_targets(&content)`
2. Extract target: `[[Target]]` → "Target", `[[Target|Display]]` → "Target"
3. Resolve target to a file ID (see resolution strategy below)
4. Insert bidirectional edges with `INSERT OR IGNORE`: `(current_file, target_file, 'wikilink')` and `(target_file, current_file, 'wikilink')`
5. Unresolved links (target not in index) are silently skipped

**Resolution strategy:**
- Exact path match: `SELECT id FROM files WHERE path = ?` (with `.md` appended if missing)
- Basename match: `SELECT id FROM files WHERE path LIKE '%/' || ? || '.md'` (case-insensitive via `COLLATE NOCASE`)
- If multiple matches, prefer the shortest path (closest to root)

### People Detection

**People folder resolution:** The `VaultProfile` (from `vault.toml`) stores `folders.people` as a folder name (e.g., `"People"`). However, in real vaults this is often nested (e.g., `03-Resources/People`). The indexer resolves it by:
1. Loading the `VaultProfile` via `Config::load_vault_profile()`
2. If `folders.people` is set, search `files` table for paths containing that folder name: `SELECT DISTINCT path FROM files WHERE path LIKE '%/' || ? || '/%'`
3. Extract the common prefix as the people folder path
4. If not configured or no matches found, skip people detection entirely (graph agent still works with wikilink edges only)

**Detection steps:**
1. Load all files matching the people folder path prefix
2. Extract person names: filename without `.md` extension (e.g., `John Nelson.md` → "John Nelson")
3. Extract aliases from frontmatter `aliases:` field (read raw file, parse frontmatter)
4. For each non-People file being indexed, scan its raw text for person name/alias matches (case-insensitive `str::to_lowercase().contains()`)
5. Create `mention` edge: `(file, person_file, 'mention')` — unidirectional (the mention is from the file, pointing to the person)

**Performance:** Person list is small (10-50 names). For 83 files and ~20 people this is negligible.

### Edge Cleanup for Incremental Indexing

Three scenarios must be handled:

**Changed files:** `delete_edges_for_file(file_id)` deletes edges in BOTH directions: `WHERE from_file = ? OR to_file = ?`. This is necessary because bidirectional wikilink insertion means both `(A → B)` and `(B → A)` exist. When A is re-indexed, both must be cleared before re-inserting A's current links. Without this, stale reverse edges would survive.

**Deleted files:** `ON DELETE CASCADE` on both foreign keys handles this automatically. When `DELETE FROM files WHERE id = ?` runs, all edges referencing that file (as `from_file` OR `to_file`) are cascaded.

**Full rebuild (`--rebuild`):** Before the edge building phase, truncate the edges table: `DELETE FROM edges`. This prevents stale edges from previous index runs when files may have lost wikilinks.

## 2. Graph Agent

A search lane that expands results from other lanes by following graph connections. All execution is synchronous and single-threaded, consistent with existing `Store` usage.

### Input

Takes the top N results from semantic + FTS lanes (before RRF fusion). These are the "seed" results.

### Algorithm

```
For each seed result (file_id, score):
    Get outgoing wikilinks (1 hop):
        For each linked file (skip if already a seed):
            expansion_score = seed_score × 0.8
            If file contains any query term (via file_contains_term, FTS5 escaped):
                Keep with expansion_score
            Else if shared_tags(seed_file, linked_file) >= 1:
                Keep with expansion_score × 0.7
            Else:
                Skip (tangential)

    Get 2-hop links (outgoing wikilinks of each 1-hop result):
        For each 2-hop file (skip if already a seed or 1-hop):
            expansion_score = seed_score × 0.5
            Same relevance filter as above

    Multi-parent merge:
        If a file is reachable from multiple seeds:
            Take the highest expansion_score (don't sum — avoids over-rewarding hub notes)

    Cap total expansions at max_expansions (default 20), sorted by score descending.
```

**Shared tags threshold:** `>= 1` shared tag (not 2). Rationale: with a typical vault having fine-grained tags, even 1 shared tag indicates topical relation. If the graph lane produces too many low-quality expansions, the RRF weight (0.8) and relevance filter ensure they rank below direct hits.

**Empty graph:** When the edges table is empty (no wikilinks in vault), `graph_expand` returns an empty `Vec`. The 3-lane RRF degrades gracefully to 2-lane — an empty lane contributes nothing to fusion scores.

**People detection disabled:** When no People folder is configured, `mention` edges are simply not created. `graph_expand` still works with wikilink edges only.

### Output

Returns `Vec<RankedResult>` compatible with the existing RRF fusion.

### Implementation: `src/graph.rs`

```rust
/// Expand search results by following graph connections.
/// Seeds are the top results from semantic + FTS lanes.
```

## 3. Three-Lane RRF Integration

### Changes to `search.rs`

New flow:
1. Run semantic lane → `Vec<RankedResult>` (seeds)
2. Run FTS lane → `Vec<RankedResult>` (seeds)
3. Combine seeds: `[&semantic_results, &fts_results].concat()` (deduplicated by file)
4. Run `graph_expand(store, &combined_seeds, query, 2, 20)` → `Vec<RankedResult>`
5. `rrf_fuse([("semantic", &semantic, 1.0), ("fts", &fts, 1.0), ("graph", &graph, 0.8)], RRF_K)`

### `--explain` Changes

Add `detail: Option<String>` field to `LaneContribution` in `fusion.rs`. Update `format_explain` to append detail when present:

```
graph: rank #1, raw 0.42 (1-hop from BRE-2579), +0.0131
```

The graph agent sets `detail` to `"{hop_depth}-hop from {seed_file}"` for each expansion.

### `engraph status` Changes

Add edge counts to `StoreStats` and the status output:

```

Files:      83
Chunks:     530
Edges:      511 (464 wikilinks, 47 mentions)
...
```

## 4. CLI: `engraph graph`

### Subcommands

```
engraph graph <file_or_docid>    Show connections for a note
engraph graph stats              Vault graph overview
```

### Input Resolution

`<file_or_docid>` is resolved in this order:
1. If starts with `#` and is 7 chars (# + 6 hex), look up via `store.get_file_by_docid()`
2. Exact path match against `files.path`
3. Basename match (case-insensitive, without `.md`)
4. If no match found, print error and exit

## 5. Indexer Changes

### Edge Building Phase

After the existing serial write phase (insert files + chunks + FTS), add a new edge building phase. This runs after ALL files are written to the store (so link targets can be resolved against the complete file set).

**Wrap in a single SQLite transaction** for performance (hundreds of INSERT OR IGNORE statements):

```rust
// Pseudocode for edge building phase
store.begin_transaction()?;

if rebuild {
    store.clear_edges()?;
}

// For changed files being re-indexed, clear their old edges
for file_id in changed_file_ids {
    store.delete_edges_for_file(file_id)?;
}

// Build edges for all files being indexed (new + changed)
for (file_path, content, file_id) in &files_to_index {
    let targets = extract_wikilink_targets(&content);
    for target_name in targets {
        if let Some(target_id) = resolve_link_target(&store, &target_name)? {
            store.insert_edge(file_id, target_id, "wikilink")?;  // outgoing
            store.insert_edge(target_id, file_id, "wikilink")?;  // incoming (backlink)
        }
    }
}

// People detection (if configured)
if let Some(people_folder) = people_folder_path {
    let people = load_people_entities(&store, &people_folder)?;
    for (file_path, content, file_id) in &files_to_index {
        if is_in_people_folder(file_path, &people_folder) { continue; }
        for person in &people {
            if content_mentions_person(&content, person) {
                store.insert_edge(file_id, person.file_id, "mention")?;
            }
        }
    }
}

store.commit_transaction()?;
```

### Transaction Support

Add to `Store`:
```rust
pub fn begin_transaction(&self) -> Result<()>  // PRAGMA foreign_keys already ON; BEGIN
pub fn commit_transaction(&self) -> Result<()>  // COMMIT
```

Or use rusqlite's `Transaction` type if cleaner.
